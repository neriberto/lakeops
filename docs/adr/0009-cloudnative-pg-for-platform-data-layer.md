## Title

Adopt CloudNativePG operator + `Cluster` `postgresql-dev` for the platform data layer; supersede ADR-0007.

> **Status:** Accepted
>
> **Date:** 2026-07-08
>
> **Supersedes:** [ADR-0007](0007-bitnami-postgresql-chart-for-platform-foundation.md)

## Context

ADR-0007 ratified the Bitnami `postgresql` chart for `dev` with `stage` / `prod` deferred to RFC-0004. The user installed the CloudNativePG operator (per commit `f4cb34e` and `1967d50` adding the operator to the AppProject allowlist) and confirmed the operational direction is operator-based from day one, not chart-based.

That changes the architectural decision: declarative HA, S3 WAL archiving (against the existing SeaweedFS S3 endpoint), Prometheus-native metrics via `_postgresql` endpoints, declarative `Pooler` integration with PgBouncer, and first-class upgrade paths are all achievable on day one in `dev` rather than deferred to `prod` per RFC-0004. The Bitnami chart path is therefore obsolete for the platform data layer — keeping a parallel chart next to an operator-managed `Cluster` would create two controllers, two credential sets, and two reconciliation loops for the same workload.

## Decision

We adopt the CloudNativePG operator plus a plain `Cluster` resource for the platform data layer, starting in `dev`. The operator install lands via the existing `cloudnative-pg` element of `apps/dev/appset.yaml` at `version: 0.29.0`; the `Cluster` resource is committed at `apps/cloudnative-pg/clusters/postgresql.yaml` and deployed by a standalone ArgoCD `Application` at `apps/cloudnative-pg/clusters/application.yaml`.

`spec.postgresqlVersion` is pinned to `"17"` (the current stable PostgreSQL major). The wrapper-chart pattern from the original planning pass is rejected: a single `Cluster` CR is plain YAML, and adding a Helm wrapper around a single resource adds complexity (in-repo Helm repo, chart versioning, two-file synchronization) that doesn't pay back.

ADR-0007 is superseded. The Bitnami `postgresql` ApplicationSet element is removed from `apps/dev/appset.yaml`; the `https://charts.bitnami.com/bitnami` URL is removed from `apps/appprojects/values.yaml` `sourceRepos`; the Bitnami values file at `apps/values/dev/postgresql.yaml` is repurposed as a documentation-of-contract file (per-env `Cluster.spec` overrides, not consumed by ArgoCD).

## Consequences

- `apps/cloudnative-pg/clusters/postgresql.yaml` carries the `Cluster` resource; ArgoCD deploys it via the standalone `Application` at `apps/cloudnative-pg/clusters/application.yaml`. The standalone Application is picked up by an additional entry appended to `apps/bootstrap/dev.yaml` (the "tiny entry" that extends the existing dev bootstrap to a multi-document YAML).
- The operator install is already running: the `cloudnative-pg` element of `apps/dev/appset.yaml` at `version: 0.29.0` deploys the operator into `cloudnative-pg-dev`. Bumping from the prior `0.26.0` is a single field edit; the ArgoCD template interpolates `{{ .version }}` into `targetRevision`, so the new value is picked up on next reconcile.
- The `Cluster`'s `bootstrap.initdb.secret.name: postgresql-credentials` references the SealedSecret-derived `Secret` from ADR-0008 (Track C — Sealed Secrets controller install). Until Track C lands, the `Cluster`'s `lake` user cannot bootstrap; the operator generates a random password as a fallback. This is acceptable for `dev` but blocks downstream consumers; the secret wiring is the critical-path follow-up.
- `apps/values/dev/postgresql.yaml` is repurposed as a documentation-of-contract file (not consumed by ArgoCD), recording the per-environment overrides that would land in the `Cluster` spec for future automation.
- The Bitnami URL is removed from `apps/appprojects/values.yaml` `sourceRepos`; the three rendered AppProject files at `apps/appprojects/rendered/{infrastructure,platform,workloads}.yaml` lose the matching fourth `sourceRepos` entry each (regenerated by `bash scripts/render-appprojects.sh`).
- SPEC-0006 (`docs/specs/0006-cloudnative-pg-pinning-and-cluster-cr-contract.md`) tracks the pinning contract for the operator chart version, the standalone Application's `Cluster.spec` shape, and the upgrade procedure. SPEC-0005 (the Bitnami pinning spec) is deleted as obsolete.
- `stage` and `prod` Cluster specs will follow the same pattern — one per-environment file alongside the operator install — when those environments activate per RFC-0004. The HA configuration (3 instances + replication slots) is a per-env config item deferred to RFC-0004 multi-cluster HA work.

## Alternatives Considered

- **Bitnami `postgresql` chart** ([ADR-0007](0007-bitnami-postgresql-chart-for-platform-foundation.md) — superseded). With the CloudNativePG operator already installed and watching `Cluster` CRs, the Bitnami chart becomes parallel infrastructure: a second controller, a second set of credentials, two PostgreSQL instances competing for the same workload consumers. The maintenance cost outweighs the familiarity benefit.
- **Zalando postgres-operator** ([github.com/zalando/postgres-operator](https://github.com/zalando/postgres-operator)). Deferred to RFC-0004. Zalando has a mature failover story and a long history; CloudNativePG's declarative `Cluster` resource and first-class `kubectl` integration are equally viable for production HA. Picking one is RFC-0004's call once multi-cluster topology lands.
- **Bitnami `postgresql-ha` subchart**. Deferred to RFC-0004. Chart-based HA is operationally heavier and lacks the S3 WAL archiving integration that CloudNativePG offers out of the box (against the existing SeaweedFS S3 endpoint).
- **Wrapper Helm chart around the `Cluster` CR** (the original Option B from the planning pass). Rejected: requires either an in-repo Helm chart index (e.g., GitHub Pages) or a different Application source shape. A single plain `Application` with a `path:` source is simpler and matches the lakeops GitOps pattern more cleanly for a single `Cluster` resource.
- **Embedded per-workload PostgreSQL** (CloudNativePG or similar inside each consuming namespace). Rejected: duplicates operational work, defeats the purpose of a shared `platform` AppProject, and creates backup sprawl. Cross-reference [RFC-0005](../rfc/0005-initial-workload-rollout-sequence.md) §Alternatives Considered.
