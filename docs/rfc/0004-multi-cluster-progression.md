## Title

Multi-cluster ArgoCD topology for `stage` and `prod`

> **Status:** Proposed
>
> **Date:** 2026-07-07
>
> **Author(s):** lakeops maintainers

## Motivation

Every Application generated by an ApplicationSet targets
`server: https://kubernetes.default.svc` today (see
[`apps/dev/appset.yaml`](../../apps/dev/appset.yaml) and the stage and prod
siblings). This is the in-cluster ArgoCD endpoint and is correct only
because every environment currently runs on a microk8s instance that ArgoCD
shares. Once stage and prod move to dedicated clusters — EKS for prod,
microk8s or EKS for stage — `kubernetes.default.svc` no longer points at
the right API server and the ApplicationSet topology has to change.

The single-cluster topology is also a single point of failure: an ArgoCD
control-plane outage halts reconciliation across every environment
simultaneously. A multi-cluster topology with a central "management" ArgoCD
keeps a single control plane that survives cluster failures independently.

## Proposal

Adopt a **hub-and-spoke** topology:

```text
                    management cluster
                    +-----------------------------+
                    |  ArgoCD (central)           |
                    |  - registers spokes         |
                    |  - hosts ApplicationSets    |
                    +-----------------------------+
                                  |
                +-----------------+-----------------+
                |                 |                 |
        dev cluster        stage cluster      prod cluster
        (spoke)            (spoke)            (spoke)
```

### Components

1. **Management cluster.** A small, dedicated Kubernetes cluster (EKS or
   microk8s) whose only workload is ArgoCD. Holds the source of truth
   ApplicationSets and the spoke cluster registrations.

2. **Spoke clusters.** The clusters where workloads actually run. Dev,
   stage, and prod are each a separate spoke. The dev spoke can stay on
   microk8s for local development; the prod spoke should be EKS or a
   similarly managed offering.

3. **Spoke registration.** Each spoke is registered with the management
   ArgoCD via `argocd cluster add`. Registration options:

   - **EKS kubeconfig.** A long-lived kubeconfig stored in ArgoCD's
     management cluster. Simplest but the kubeconfig is a powerful secret
     and rotation requires updating every spoke.
   - **IAM Role for Service Accounts (IRSA).** ArgoCD authenticates to
     EKS via AWS IAM. No kubeconfig to manage. Requires an OIDC provider
     on the EKS cluster and a trust policy on the role. The right answer
     for EKS spokes.
   - **Plain secret in the management cluster.** Default option for
     non-EKS spokes. Same caveats as the EKS kubeconfig.

4. **ApplicationSets that emit one Application per spoke.** A single
   ApplicationSet generator produces one Application per `(component,
   cluster)` pair. Today the generator is keyed by environment (one appset
   per `apps/{dev,stage,prod}/`); the multi-cluster form adds a second
   dimension:

   ```yaml
   # apps/workloads/appset.yaml (illustrative)
   generators:
     - list:
         elements:
           - chart: trino
             namespace: trino-dev
             project: workloads
             clusterName: dev
             syncWave: 2
           - chart: trino
             namespace: trino-stage
             project: workloads
             clusterName: stage
             syncWave: 2
   ```

   The ApplicationSet becomes a single source of truth for *what runs
   where* across clusters.

5. **Per-cluster registration secrets.** Stored as Kubernetes `Secret`
   resources in the management cluster's ArgoCD namespace, owned by a
   dedicated AppProject (for example, `argocd-spokes`) with tight
   RBAC. See [RFC-0002](0002-secrets-management.md) for the broader
   secrets strategy.

## Drawbacks

- **Operational surface area grows.** Three clusters to monitor, upgrade,
  and back up instead of one. The management cluster is a new dependency
  for the platform's bootstrap.
- **Spoke registration secrets are high-value targets.** A leaked
  management-cluster secret can take over every spoke. The `argocd-spokes`
  AppProject must apply the same defense-in-depth principles as
  [ADR-0004](../adr/0004-defense-in-depth-appproject-config.md).
- **Cross-cluster networking.** Workloads in one cluster reaching services
  in another (for example, Trino in prod reading from PostgreSQL in a
  shared services cluster) require either a service mesh, VPC peering, or
  external service endpoints. The proposal does not address this; it is
  out of scope.
- **Multi-cluster ApplicationSets duplicate state.** Each spoke gets its
  own copy of the Application, values file, and AppProject. The list
  generator can become large; a matrix generator (combining `cluster` and
  `chart` dimensions) is the natural next step once the list grows past
  ~20 entries.

## Alternatives Considered

- **Per-cluster ArgoCD (no management cluster).** Each cluster runs its
  own ArgoCD instance. Maximum isolation, but every cluster needs its own
  ApplicationSet source of truth, and a drift between clusters is hard to
  detect. Rejected for now because the management cluster's operational
  cost is much smaller than the cross-cluster drift detection cost.
- **Single ArgoCD with multiple `destination.server` entries in
  AppProjects.** This is the in-tree equivalent of hub-and-spoke without
  the spoke registration. Works when ArgoCD runs in a cluster with
  network reachability to every spoke's API server (for example, a VPC
  peered into each spoke). Closer to the current single-cluster model and
  a smaller migration step; can be a stepping stone toward the full
  hub-and-spoke proposal.
- **Cluster-api + ArgoCD.** Cluster lifecycle and workload lifecycle in
  one control plane. Powerful but a much bigger investment than the
  platform needs at this stage.

## Open Questions

- **Which cluster becomes the management cluster?** EKS is the natural
  choice for prod-like availability, but the management cluster does not
  need prod-like resources. A small dedicated cluster (microk8s on a
  small VM, or a minimal EKS node group) may be sufficient.
- **EKS kubeconfig vs IRSA vs plain secret?** For EKS spokes, IRSA is
  the right answer. For microk8s spokes, kubeconfig or a token-based
  registration. The choice depends on which spoke types are deployed
  first.
- **Interaction with RFC-0001 allowlist uniformity?** The destination
  allowlist uniformity invariant (see
  [RFC-0001](0001-destination-allowlist-uniformity.md)) currently assumes
  a single shared in-cluster endpoint. Multi-cluster requires that the
  allowlist enumerate every spoke's server URL. RFC-0001's `clusters`
  map proposal is the natural data shape for this — the two RFCs
  should land together.
